# Architecture Overview

## Project Structure

```
cmd/                          # Application entrypoints
  server/main.go              #   Main server binary
proto/services/<svc>/v1/      # Protobuf service definitions (API contracts)
handlers/<svc>/               # Go handler implementations (YOUR business logic)
  service.go                  #   Handler methods
  authorizer.go               #   Custom authorization (yours to edit)
  authorizer_gen.go           #   Generated RBAC (regenerated — do not edit)
  handlers_crud_gen.go        #   Generated CRUD handlers (regenerated)
frontends/<name>/             # Next.js frontends
  src/app/                    #   App Router pages and layouts
  src/hooks/                  #   Generated + custom hooks
  src/lib/                    #   Utilities and Connect client setup
gen/                          # ALL generated code — NEVER hand-edit
  go/                         #   Go stubs (protoc-gen-go, protoc-gen-connect-go)
  ts/                         #   TypeScript clients
internal/db/                  # Database layer (YOU own this)
  types.go                    #   Entity types (aliases or concrete structs)
  <entity>_orm.go             #   CRUD functions per entity
  mappers.go                  #   Proto ↔ DB type mappers (when they diverge)
internal/<name>/              # Internal Go packages with interface contracts
  contract.go                 #   Interface definition
  <name>.go                   #   Implementation
pkg/app/                      # Application wiring
  bootstrap.go                #   Generated service registration — DO NOT EDIT
  setup.go                    #   Custom wiring — YOUR hook (//forge:allow)
  testing.go                  #   Test harness for integration tests
pkg/middleware/               # HTTP/Connect middleware (generated)
pkg/config/                   # Config struct + loader
db/migrations/                # SQL migrations — THE schema source of truth
db/queries/                   # SQL query definitions
deploy/kcl/<env>/             # KCL deployment manifests per environment
  main.k                      #   Hand-edited per-env entry point
  config_gen.k                #   GENERATED per-env env-var groups
config.<env>.yaml             # Optional sibling per-env config (prod typical)
e2e/                          # End-to-end tests
forge.yaml                    # Project config: services, ports, frontends, packs
forge_descriptor.json         # Proto descriptor data (generated)
```

## Generated vs Hand-Written

| Forge generates (safe to regenerate) | You own (Forge never touches) |
|--------------------------------------|-------------------------------|
| `gen/` — Go stubs, TS clients, mocks | `handlers/<svc>/service.go` — business logic |
| `pkg/app/bootstrap.go` — service wiring | `pkg/app/setup.go` — custom wiring |
| `pkg/middleware/` — HTTP/Connect middleware | `internal/db/` — entity types, ORM functions |
| `handlers/<svc>/*_gen.go` — CRUD, authorizer | `handlers/<svc>/authorizer.go` — custom auth |
| Frontend hooks (`*-hooks.ts`) | `db/migrations/` — schema source of truth |
| `forge_descriptor.json` | `db/queries/` — SQL queries |
| `frontends/<name>/src/lib/connect.ts` | `internal/<pkg>/` — internal packages |

**Rule of thumb**: If it has `_gen` in the name or lives in `gen/`, it's regenerated. Everything else is yours.

### Three precise classes (the unambiguous version)

| Class | Signal | Behaviour |
|-------|--------|-----------|
| **Pure forge-space** | `_gen` filename + `// Code generated by forge. DO NOT EDIT.` header + `// Source: …` pointer | Regenerated on every `forge generate`. User edits are blown away. Listed in `.gitignore` (not committed). |
| **Scaffold-with-placeholders** | `_gen` filename + body contains `// FORGE_SCAFFOLD: <what to do>` | Regenerated as long as any marker remains. The moment every `FORGE_SCAFFOLD:` line is removed the file becomes user-owned and forge stops overwriting it. Currently used for `handlers_crud_gen_test.go` and `handlers_crud_integration_test.go`. |
| **Pure user-space** | No `_gen` suffix, no DO NOT EDIT header | Scaffolded once, never overwritten. May start with a `// FORGE_SCAFFOLD: …` line on placeholder bodies (e.g. `setup.go`'s "wire your custom dependencies here") to flag the user's job. |

`forge lint --scaffolds` enforces this:

- A `FORGE_SCAFFOLD:` marker in committed code is a build-gating error
  (`scaffold-not-customized`) — either fill in the placeholder and remove
  the marker, or remove the file from the commit.
- An `_gen.go` file missing the canonical `// Code generated by forge. DO
  NOT EDIT.` header is a build-gating error (`gen-missing-header`).
- An `_gen.go` file missing the `// Source: …` pointer is a warning
  (`gen-missing-source`).

## The Generate Pipeline

```
proto/services/<svc>/v1/<svc>.proto
  → protoc-gen-forge --mode=descriptor → forge_descriptor.json
  → protoc-gen-forge --mode=orm        → *.pb.orm.go (ORM code)
  → protoc-gen-go + protoc-gen-connect-go → gen/ stubs

internal/*/contract.go (Go interfaces)
  → forge generate → mock_gen.go, middleware_gen.go, tracing_gen.go

forge_descriptor.json + handlers/<svc>/
  → forge generate → handlers_crud_gen.go (thin shim → forge/pkg/crud), authorizer_gen.go

gen/ts/ (TypeScript clients)
  → forge generate → frontends/<name>/src/hooks/*-hooks.ts
```

Running `forge generate` is always safe. It only touches infrastructure — never your handlers, DB layer, or business logic.

## Custom Wiring in setup.go

`pkg/app/bootstrap.go` is generated and auto-registers services, workers, and internal packages. **Never edit it.**

`pkg/app/setup.go` is yours. Use it to wire custom dependencies — database handles, external clients, feature flags, anything `bootstrap.go` can't know about:

```go
// pkg/app/setup.go
func Setup(app *App) error {
    // Wire custom dependencies here
    app.UserService.DB = app.Pool
    app.UserService.EmailClient = ses.NewClient(app.Config.AWS)
    return nil
}
```

`setup.go` is marked with `//forge:allow` and will never be overwritten.

## Test Harness in testing.go

`pkg/app/testing.go` provides helpers for integration tests — bootstrapping a real app with a test database, authenticated clients, and cleanup:

```go
func TestCreateUser(t *testing.T) {
    harness := app.NewTestHarness(t)
    client := harness.AuthenticatedClient(t, "user-1", "admin")
    // ... test with real DB and middleware
}
```

The harness runs migrations, seeds data, and tears down after each test.

## Files NOT to Edit

These are regenerated by `forge generate` — your changes will be overwritten:

- `gen/` — All generated Go and TypeScript code
- `pkg/app/bootstrap.go` — Service registration and wiring
- `pkg/middleware/` — HTTP/Connect middleware
- `handlers/<svc>/*_gen.go` — Generated CRUD handlers and authorizers
- `frontends/<name>/src/hooks/*-hooks.ts` — Generated React Query hooks
- `frontends/<name>/src/lib/connect.ts` — Connect transport setup
- `forge_descriptor.json` — Proto descriptor data

## Config + environments

Runtime config has two layers:

1. **Schema** lives in `proto/config/v1/config.proto` as an `AppConfig`
   message. Each field carries a `(forge.v1.config)` annotation —
   `env_var`, `flag`, `default_value`, `description`, plus the deploy-time
   knobs `sensitive` (project to a Kubernetes Secret) and `category`
   (group fields together for KCL emission).
2. **Per-env values** live under `environments[<env>].config` in
   forge.yaml *or* in a sibling file `config.<env>.yaml` (sibling wins on
   key collision). Inline is good for dev/staging; sibling is good for
   prod where most values are secret refs.

`forge generate` projects both into `deploy/kcl/<env>/config_gen.k`,
emitting `EnvVar { value = ... }` for non-sensitive scalars and
`EnvVar { secret_ref = ... }` for sensitive ones. The hand-edited
`main.k` for each env imports the generated module.

`forge run --env <env>` exports the same per-env values as process
env-vars to the running binary, skipping sensitive fields (use direnv).
`forge deploy <env>` passes non-sensitive scalars to KCL via `-D
key=value`.

## Operators and CRDs

Forge supports controller-runtime operators as a first-class component
type. The pattern is the same as `forge/pkg/contractkit`: the heavy
lifting lives in a runtime library, and codegen emits a thin per-CRD
shim.

### Library — `forge/pkg/controller`

The library carries:

- `Reconciler[T client.Object]` — generic typed base. Owns the
  fetch / NotFound / finalizer-add / finalizer-cleanup / dispatch
  lifecycle and exposes a small `Run(ctx, req, blank, reconcileFn,
  finalizeFn)` entry point that the per-CRD shim calls from its
  controller-runtime `Reconcile` method.
- `Result` + `Done() / Requeue(after) / Stop()` constructors —
  thin wrappers over `ctrl.Result` so reconcile code reads as
  `return controller.Requeue(5*time.Second), nil`.
- Predicates: `SkipDeletion()`, `HasAnnotation(key)`,
  `HasLabel(selector)`, `AnnotationChanged(key)`. Compose them via
  `SetupOptions.ExtraPredicates`.
- `ClusterClientManager` — multi-cluster client cache lifted from
  control-plane-next. Safe for concurrent use.
- `Backoff{Initial, Max, Factor}.Next(attempt)` — capped-exponential
  helper for reconcilers that track per-object retry counts.
- `controllertest.New(t, opts...)` — envtest harness that calls
  `t.Skip` cleanly when binaries are missing (no
  `KUBEBUILDER_ASSETS`, no `setup-envtest` cache).

### Commands

```
forge add operator <name>                                  # operator binary + manager wiring
forge add operator <name> --with-placeholder-crd           # legacy combined scaffold
forge add crd <Name> --shape state-machine|config|composite --operator <op>
```

`forge add operator` creates `operators/<name>/{doc.go,operator.go}`
plus the bootstrap wiring. The operator package only owns
manager-level deps; per-CRD reconcilers are added separately via
`forge add crd`.

`forge add crd <Name>` emits three files:

- `api/<version>/<name>_types.go` — Spec / Status / DeepCopy.
- `operators/<operator>/<name>_controller.go` — thin shim that
  embeds `controller.Reconciler[*<api>.<Name>]` and implements
  `ReconcileSpec(ctx, obj)` + `FinalizeSpec(ctx, obj)`.
- `operators/<operator>/<name>_controller_test.go` — fake-client
  unit test (envtest is opt-in via `controllertest.New`).

A shared `api/<version>/groupversion.go` is created on the first CRD
add and re-used by subsequent ones — multiple CRDs in the same
version package coexist via `init()`-time `SchemeBuilder.Register`.

### Shapes

- **state-machine** — Spec.State drives the loop through observable
  phases. Generated controller has `reconcileRunning /
  reconcileSuspended / reconcileDeleting` helpers and a switch over
  `Spec.State`. Spec gains a `State` field and Status gains
  `Phase`, `Message`, `ObservedGeneration`.
- **config** — declarative-only. Spec describes a configuration to
  project into a runtime store; no observable state. Generated
  controller has a single `ReconcileSpec`.
- **composite** — Spec owns sub-resources whose lifetime is coupled
  to the parent. Generated controller is structured around `ensure*`
  helpers and a periodic requeue.

## Binary modes

`forge.yaml` `binary:` selects how the project packages its
service-handlers across Go binaries. Two modes are supported:

### `binary: per-service` (default)

The canonical layout. Forge ships **one binary per project** (`cmd/main.go`)
that exposes the full surface area through cobra subcommands —
`server`, `worker`, `operator`, `db`, `version`, etc. The `server`
subcommand accepts a positional list of service names so the same
binary can run a subset of registered services in any given pod:

```
./<project> server admin-server         # mount only admin-server's handlers
./<project> server admin-server billing # mount admin-server + billing
./<project> server                      # mount everything (default)
```

`Bootstrap` constructs all services as cheap structs, then
`BootstrapOnly` mounts only the requested ones on the mux. KCL
deploys emit one `Application` per service.

### `binary: shared`

One Go binary, **one cobra subcommand per service**. Each service is
its own first-class subcommand:

```
./<project>                         # --help lists api / billing / worker / ...
./<project> api                     # boots ONLY the api service's deps + handlers
./<project> worker                  # boots ONLY the worker
./<project> server api billing      # canonical multi-service form still works
```

`BootstrapOnly` switches to lazy per-service construction in this
mode — calling `BootstrapOnly(mux, logger, cfg, []string{"api"})`
constructs ONLY the api service's dependency graph and skips every
other service's constructors. This is the runtime substrate that
makes `MultiServiceApplication` deploys (see the `deploy` skill)
genuinely scope-down per-pod.

KCL deploy in this mode emits one `MultiServiceApplication` block
(one `image:`, N `SubCommandService` entries) instead of N
`Application` blocks. Cuts image build/push from N to 1.

Pick this mode when:

- You have 3+ services that share substantial `internal/` libraries.
- CI image build/push time dominates cycle time (linear in service
  count for per-service mode).
- Operationally, services already ship together (you don't release
  one without the others).

Stick with `per-service` when services have genuinely divergent
runtime dependencies (CGO + libpq for one, pure Go for another), or
when independent per-service image SHAs are a hard requirement of
your release engineering. See the `migration/v0.x-to-binary-shared`
skill for migration steps and trade-offs.

`forge new <project> --binary shared --service A --service B` opts
in at scaffold time. Existing projects opt in by setting `binary:
shared` in forge.yaml and running `forge generate`.

## Key Commands

| Command | When to use |
|---------|------------|
| `forge generate` | After any proto or contract change |
| `forge build` | Verify everything compiles |
| `forge run` | Start the full dev stack |
| `forge run --env <env>` | Start the dev stack with per-env config layered in |
| `forge test` | Run all tests |
| `forge deploy <env>` | Deploy to a Kubernetes environment with per-env config |
| `forge add operator <name>` | Add a controller-runtime operator (manager binary) |
| `forge add crd <Name>` | Add a CRD + reconciler shim to an operator |
| `forge generate --explain` | Per-file provenance: which proto/contract drove each output |
| `forge audit` | Comprehensive project state snapshot (version, shape, codegen, packs, scaffolds, deps) |
| `forge audit --json` | Machine-readable audit (sub-agent-friendly) |
| `forge map` | Annotated project tree — every file labelled user-owned vs forge-space |
| `forge map --filter handlers/` | Subtree-filtered map |

## Introspection Workflow

When orienting in a forge project, prefer the introspection commands over
ad-hoc grep:

1. `forge audit` — overall health, drift signals, version pin.
2. `forge map --depth 2` — high-level project shape.
3. `forge map --filter <subtree>` — drill into the area in scope.
4. After codegen runs, `forge generate --explain` — see why each output was rewritten or skipped.

JSON output everywhere lets sub-agents chain calls
(`forge audit --json | jq '.categories.codegen.details.user_edited_gen_files'`).

## Rules

- Never hand-edit anything under `gen/` or any `*_gen.go` file. Fix the proto or contract, then regenerate.
- `bootstrap.go` is regenerated — all custom wiring goes in `setup.go`.
- `forge generate` is always safe — it only touches infrastructure.
- One service per proto package. One handler directory per service.
- `forge generate` does NOT touch the DB layer — you own `internal/db/` and `db/migrations/`.
