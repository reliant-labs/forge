## 1. K8s Operator Component Type

### Problem
Many production Go systems include Kubernetes controllers/operators. These follow an extremely formulaic pattern (controller-runtime reconciler), but Forge has no scaffolding for them. Teams hand-wire CRDs, RBAC, reconcilers, and status management every time.

### What to Build
`forge add operator <name>` — scaffolds a Kubernetes controller with:

#### Generated Files
```
operators/<name>/
├── controller.go          # Reconciler struct, Deps, SetupWithManager()
├── controller_test.go     # envtest-based reconciler tests
├── types.go               # CRD spec/status types (Go structs, not proto)
└── rbac.go                # RBAC annotations as comments (kubebuilder markers)
```

Plus:
```
api/<name>/v1/
├── types.go               # CRD Go types with kubebuilder markers
├── groupversion_info.go   # SchemeBuilder registration
└── zz_generated.deepcopy.go  # (generated by controller-gen)
```

#### Templates to Create
`internal/templates/operator/`

**controller.go.tmpl** — The reconciler:
```go
package {{.Name}}

import (
    "context"
    "log/slog"

    ctrl "sigs.k8s.io/controller-runtime"
    "sigs.k8s.io/controller-runtime/pkg/client"
    apiv1 "{{.Module}}/api/{{.Name}}/v1"
)

type Deps struct {
    Logger *slog.Logger
    Client client.Client
}

type Controller struct {
    deps Deps
}

func New(deps Deps) *Controller {
    return &Controller{deps: deps}
}

func (c *Controller) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    logger := c.deps.Logger.With("name", req.NamespacedName)

    var obj apiv1.{{.TypeName}}
    if err := c.deps.Client.Get(ctx, req.NamespacedName, &obj); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }

    logger.Info("reconciling")
    // TODO: implement reconciliation logic

    return ctrl.Result{}, nil
}

func (c *Controller) SetupWithManager(mgr ctrl.Manager) error {
    return ctrl.NewControllerManagedBy(mgr).
        For(&apiv1.{{.TypeName}}{}).
        Complete(c)
}
```

#### CLI Integration
- New command in `internal/cli/add.go`: `newAddOperatorCmd()`
- New generator: `internal/generator/operator_gen.go`
- Config tracking: add `Operators []OperatorConfig` to `ProjectConfig`
- `forge.yaml`:
  ```yaml
  operators:
    - name: workspace
      group: reliant.io
      version: v1
  ```

#### Bootstrap Integration
Operators need a different entry point than HTTP services. Two options:

**Option A (recommended):** Separate `cmd-operator.go.tmpl` subcommand:
```
myproject operator [name...]    # runs operator(s) with controller-runtime manager
```
This keeps HTTP services and K8s controllers in the same binary but with separate startup paths.

**Option B:** Integrate into `Bootstrap()` — manager runs alongside HTTP server. Simpler but couples lifecycles.

#### Generate Pipeline
Add a step to `runGeneratePipeline()`:
- Run `controller-gen` for CRD manifests and deepcopy (if operators exist)
- Output CRD YAMLs to `deploy/crds/`

#### Dependencies
- `sigs.k8s.io/controller-runtime`
- `k8s.io/apimachinery`
- `sigs.k8s.io/controller-tools` (for controller-gen)

#### Implementation Steps
1. Create template files in `internal/templates/operator/`
2. Create CRD type templates in `internal/templates/operator-api/`

## 2. Multi-Tenancy Primitives — ✅ DONE

> **Status:** Implemented. CRUD codegen now supports `tenant_key` annotation for mandatory tenant-scoped queries. Tenant context middleware extracts tenant identity from JWT claims. Generated ORM methods require a `tenantID` parameter when `tenant_key: true` is set, making it a compile-time error to forget tenant scoping.

### Problem
Most SaaS backends need tenant isolation — scoping data access by organization/tenant. This is repetitive, error-prone, and a security-critical concern. Getting it wrong means data leaks between tenants.

### What to Build

#### 2a. Tenant Context Middleware
A Connect interceptor that extracts tenant identity from the auth context and makes it available to handlers.

**New template:** `internal/templates/middleware/tenant.go.tmpl`
```go
package middleware

import (
    "context"
    "connectrpc.com/connect"
)

type tenantContextKey struct{}

type TenantInfo struct {
    TenantID string
    Role     string
}

func TenantFromContext(ctx context.Context) (TenantInfo, bool) {
    t, ok := ctx.Value(tenantContextKey{}).(TenantInfo)
    return t, ok
}

func MustTenantFromContext(ctx context.Context) TenantInfo {
    t, ok := TenantFromContext(ctx)
    if !ok {
        panic("tenant not in context — is TenantInterceptor configured?")
    }
    return t
}

// TenantInterceptor extracts tenant info from auth claims and adds it to context.
// tenantClaim is the JWT claim key that holds the tenant/org ID (e.g., "org_id").
func TenantInterceptor(tenantClaim string) connect.UnaryInterceptorFunc {
    return func(next connect.UnaryFunc) connect.UnaryFunc {
        return func(ctx context.Context, req connect.AnyRequest) (connect.AnyResponse, error) {
            claims, ok := ClaimsFromContext(ctx)
            if !ok {
                return next(ctx, req)
            }

            tenantID, _ := claims[tenantClaim].(string)
            if tenantID == "" {
                return nil, connect.NewError(connect.CodePermissionDenied, fmt.Errorf("missing tenant claim %q", tenantClaim))
            }

            info := TenantInfo{
                TenantID: tenantID,
                Role:     claims["role"].(string),
            }
            ctx = context.WithValue(ctx, tenantContextKey{}, info)
            return next(ctx, req)
        }
    }
}
```

#### 2b. Tenant-Scoped ORM Queries
Extend `protoc-gen-forge-orm` to recognize a tenant field annotation:

```protobuf
message Workspace {
    option (forge.orm.v1.message) = { table: "workspaces" };

    string id = 1 [(forge.orm.v1.field) = { primary_key: true }];
    string org_id = 2 [(forge.orm.v1.field) = { tenant_key: true, not_null: true }];
    string name = 3;
}
```

When `tenant_key: true` is set, generated CRUD methods gain a **required** `tenantID` parameter:
```go
// Generated — tenant scoping is mandatory, not optional
func ListWorkspaces(ctx context.Context, db *sql.DB, tenantID string) ([]*Workspace, error) {
    // WHERE org_id = $1 is always applied
}

func GetWorkspaceByID(ctx context.Context, db *sql.DB, tenantID string, id string) (*Workspace, error) {
    // WHERE id = $1 AND org_id = $2
}
```

This makes it **impossible** to forget tenant scoping — it's a compile-time requirement.

#### 2c. Tenant-Aware Test Harness
Extend `bootstrap_testing.go.tmpl` to set up isolated test tenants:
```go
func (h *TestHarness) WithTenant(tenantID string) context.Context {
    return context.WithValue(h.Ctx, tenantContextKey{}, TenantInfo{TenantID: tenantID})
}
```

#### Implementation Steps
1. Add `tenant.go.tmpl` to `internal/templates/middleware/`
2. Add `tenant_key` field option to `pkg/proto/forge/orm/v1/options.proto`
3. Update `protoc-gen-forge-orm` to detect tenant_key and modify generated CRUD signatures
4. Update `bootstrap.go.tmpl` to optionally include `TenantInterceptor` in the interceptor chain
5. Add `TenantConfig` to `ProjectConfig`:
   ```yaml
   auth:
     provider: supabase
     tenant_claim: "org_id"   # NEW — enables tenant middleware
   ```
6. Update `bootstrap_testing.go.tmpl` with `WithTenant()` helper
7. Add `forge.yaml` flag: when `auth.tenant_claim` is set, tenant middleware is generated

## 3. Proto-Annotated Interceptor Wiring — ✅ DONE (Auth/RBAC)

> **Status:** Auth and RBAC annotations are now functional. `protoc-gen-forge-orm` reads `auth_required` and role annotations from proto method options and generates a per-method RBAC policy map. The generated auth interceptor consults this map at runtime. Rate-limit and cache annotation wiring remain planned.

### Problem
Forge already has proto annotations for auth (`auth_required`), rate limiting (`rate_limit`), and caching (`cache`) in `method.proto`, but the code generation doesn't wire these into actual interceptors automatically. The annotations exist but are decorative.

### What to Build
Make the existing proto annotations functional by generating interceptors that read them at startup.

#### 3a. Auth Annotation → Interceptor Wiring

The `method.proto` already has:
```protobuf
message MethodOptions {
    bool auth_required = 1;
    // ...
}
```

**Generate a method-level auth registry** during `forge generate`:

**New generated file:** `pkg/app/method_auth_gen.go`
```go
package app

// Auto-generated from proto annotations. DO NOT EDIT.

var methodAuthRequired = map[string]bool{
    "/services.echo.v1.EchoService/Echo": false,
    "/services.users.v1.UserService/GetUser": true,
    "/services.users.v1.UserService/ListUsers": true,
    "/services.billing.v1.BillingService/CreateCheckout": true,
}
```

Then the auth interceptor checks this map instead of requiring per-service Authorizer boilerplate for simple auth-required checks.

#### 3b. Rate Limit Annotation → Interceptor

The `method.proto` already has:
```protobuf
message RateLimitOptions {
    int32 requests_per_second = 1;
    int32 burst = 2;
}
```

**Generate a rate limit config registry:** `pkg/app/method_ratelimit_gen.go`
```go
var methodRateLimits = map[string]RateLimitConfig{
    "/services.billing.v1.BillingService/CreateCheckout": {RequestsPerSecond: 10, Burst: 20},
}
```

**New middleware template:** `internal/templates/middleware/ratelimit.go.tmpl`
- Uses `golang.org/x/time/rate` for token bucket per-method
- Keyed by client IP or tenant ID (configurable)
- Returns `connect.CodeResourceExhausted` when exceeded

#### 3c. Cache Annotation → Interceptor

```protobuf
message CacheOptions {
    int32 ttl_seconds = 1;
    string key_template = 2;
}
```

**Generate cache config:** `pkg/app/method_cache_gen.go`  
**New middleware template:** `internal/templates/middleware/cache.go.tmpl`
- In-memory LRU cache per method (no external dependency)
- Cache key from request fields via template
- Respects TTL from annotation

#### New Generator Step

Add to `runGeneratePipeline()`:
```
Step 5c: generateInterceptorWiring()
    → Parse all service protos for method-level annotations
    → Generate method_auth_gen.go, method_ratelimit_gen.go, method_cache_gen.go
    → Only generates files for annotations that are actually used
```

#### Implementation Steps
1. Create `internal/codegen/annotation_parser.go` — extracts method annotations from parsed protos
2. Create `internal/generator/interceptor_gen.go` — generates the method registries
3. Add `ratelimit.go.tmpl` to `internal/templates/middleware/`
4. Add `cache.go.tmpl` to `internal/templates/middleware/`
5. Update `cmd-server.go.tmpl` interceptor stack to use generated registries
6. Add `generateInterceptorWiring()` step to `runGeneratePipeline()`
7. Update `bootstrap.go.tmpl` to construct rate limiter and cache if annotations are present
8. Tests: add proto fixtures with annotations and verify generated output

## 4. Scheduled Jobs / Cron Worker Kind

### Problem
Forge has a `worker` component type with `Start(ctx)/Stop(ctx)` lifecycle, but no built-in pattern for **scheduled/periodic** work. Most backends have cron-like jobs: cleanup old data, sync external state, send digest emails, check expiring subscriptions.

### What to Build
A new worker kind: `forge add worker <name> --kind cron`

#### Generated Files
```
workers/<name>/
├── worker.go              # CronWorker with schedule registration
├── jobs.go                # Individual job implementations
└── worker_test.go         # Tests with manual tick
```

**worker.go.tmpl (cron kind):**
```go
package {{.Name}}

import (
    "context"
    "log/slog"
    "time"
)

type Job struct {
    Name     string
    Schedule time.Duration  // simple interval-based
    Fn       func(ctx context.Context) error
}

type Deps struct {
    Logger *slog.Logger
    Config *config.Config
    // Add your dependencies here
}

type Worker struct {
    deps    Deps
    jobs    []Job
    cancel  context.CancelFunc
    done    chan struct{}
}

func New(deps Deps) *Worker {
    w := &Worker{deps: deps, done: make(chan struct{})}
    w.registerJobs()
    return w
}

func (w *Worker) registerJobs() {
    w.jobs = []Job{
        // Register your jobs here:
        // {Name: "cleanup", Schedule: 1 * time.Hour, Fn: w.cleanup},
    }
}

func (w *Worker) Start(ctx context.Context) error {
    ctx, w.cancel = context.WithCancel(ctx)
    for _, job := range w.jobs {
        go w.runJob(ctx, job)
    }
    return nil
}

func (w *Worker) runJob(ctx context.Context, job Job) {
    ticker := time.NewTicker(job.Schedule)
    defer ticker.Stop()

    w.deps.Logger.Info("job scheduled", "job", job.Name, "interval", job.Schedule)

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            start := time.Now()
            if err := job.Fn(ctx); err != nil {
                w.deps.Logger.Error("job failed", "job", job.Name, "error", err, "duration", time.Since(start))
            } else {
                w.deps.Logger.Info("job completed", "job", job.Name, "duration", time.Since(start))
            }
        }
    }
}

func (w *Worker) Stop(ctx context.Context) error {
    if w.cancel != nil {
        w.cancel()
    }
    return nil
}
```

#### Integration with Existing Worker System
This reuses the existing `Start(ctx)/Stop(ctx)` worker lifecycle — the server template already manages workers. The cron kind just provides the scheduling loop internally.

Config tracking:
```yaml
services:
  - name: cleanup
    type: worker
    kind: cron    # NEW — distinguishes from plain workers
```

#### Implementation Steps
1. Create `internal/templates/worker-cron/` directory with templates
2. Add `--kind` flag to `forge add worker` command in `internal/cli/add.go`
3. Update `internal/generator/worker_gen.go` to select templates based on kind
4. Update `ServiceConfig` to include optional `Kind` field for workers
5. Templates reuse existing worker lifecycle — no bootstrap changes needed
