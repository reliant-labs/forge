# Forge File Ownership Audit

This document classifies every file produced by forge templates into one of
three buckets so the LLM (and human) has unambiguous signals about who owns
what:

- **Pure forge-space** — header `// Code generated by forge. DO NOT EDIT.`,
  `_gen` suffix in the filename, regenerated on every `forge generate`. User
  edits are blown away. These files are listed in `.gitignore` for the
  scaffolded project, so they are *not* committed.
- **Pure user-space** — scaffolded once by `forge new` / `forge add ...` /
  `forge package new`, never overwritten afterwards. No `_gen` suffix, no
  "DO NOT EDIT" header. The user owns the file outright.
- **Scaffold-with-placeholders** — initially scaffolded with `FORGE_SCAFFOLD:`
  markers indicating "your job here". Once the user customises (i.e. removes
  the markers), forge stops overwriting it. While the markers are present
  the file is treated as forge-owned and gets refreshed; once they're gone
  the file is user-owned.

The classification is the single biggest brittleness fix: today it's
ambiguous whether `handlers_crud_gen_test.go` (a regenerated unit-test
scaffold full of TODOs) is forge-space or user-space. The convention below
makes the answer mechanical.

## Marker convention

```go
// FORGE_SCAFFOLD: <what to do>
// Remove this comment after editing.
```

A scaffold-with-placeholders file is regenerated by forge **as long as any
`FORGE_SCAFFOLD:` comment remains**. The first time the user removes every
marker, forge transitions the file to user-space and never overwrites it
again. `forge lint --scaffolds` flags both states (markers committed and
unsigned `_gen` edits).

## Header convention for pure forge-space files

Every truly-regenerated file gets a uniform header:

```go
// Code generated by forge. DO NOT EDIT.
// Source: <where the input lives>
//
// To customize: <what to do, e.g. "remove _gen.go and create your own implementation">
```

## Lint analyzer behaviour

`forge lint --scaffolds` fires when:

1. A scaffold-with-placeholders file (or one of the above mixed-state files
   converted to one) still contains `FORGE_SCAFFOLD:` markers AND the file
   is tracked by git (i.e. the user committed the unfinished scaffold).
2. A pure forge-space `_gen` file has been hand-edited — detected by
   re-rendering the canonical template output and diffing.

It does NOT fire on a fresh `forge new` checkout: the `_gen.go` files
generated under `handlers/<svc>/` are gitignored, and a scaffold checked
out by `forge new` hasn't been committed yet, so case 1 doesn't trip. The
analyzer is therefore safe to wire into pre-commit and CI.
