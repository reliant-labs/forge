# 2. Frontend scenarios for URL-driven mock state

- Status: proposed
- Date: 2026-05-28
- Deciders: project authors

## Context and Problem Statement

Forge-generated frontends already ship with a mock transport (`src/lib/mock-transport.ts`) that intercepts every Connect RPC and returns deterministic fixture data when `VITE_MOCK_API=true` (or `NEXT_PUBLIC_MOCK_API=true`). This is sufficient to render a happy-path UI without a backend, and to drive Chrome DevTools MCP through pages that *don't* care about specific backend state.

It is **not** sufficient for the larger class of UI bugs that depend on specific server-state shapes — e.g. "the settings page misreports GitHub connection status because the parser is reading the wrong field name". To validate that class of bug today, a human has to drive through OAuth in a real browser session against a real backend, then hand control to the agent. That defeats the entire point of an agent-driven debugging loop.

We need a way to declare named server states ("scenarios") so an agent can teleport into any of them by navigating to a URL.

## Decision Drivers

- **Production code path must run.** The bug class we care about lives in the deserialization / state-derivation layer; if a scenario seeds React Query directly we skip the exact line that's broken.
- **Composes with existing fixtures.** Forge already generates deterministic per-entity mock data; scenarios should override only what they care about and let everything else fall through.
- **Typed at write-time.** Handlers are `(Request) => Response` typed against the generated proto schemas. No JSON-blob drift.
- **URL-driven.** A single `?scenario=name` query param selects the active scenario; same URL works for human exploration, Chrome MCP, and vitest tests.
- **One source of truth.** Scenarios live in source under `src/mocks/scenarios/`, are version-controlled, and the registry is regenerated when scenarios are added or removed.

## Decision Outcome

Add a thin scenarios layer on top of the existing mock transport.

### Files emitted

```
src/mocks/
├── scenario-types.ts          AUTO    interface Scenario, defineScenario helper
├── scenarios/
│   ├── index.ts               AUTO    registry — { default, github-connected, ... }
│   ├── default.ts             SEED    fallback scenario, empty handlers, no-op setup
│   ├── github-connected.ts    USER    user-authored via `forge add scenario`
│   └── …
└── mock-transport.ts          AUTO    reads ?scenario= once, picks scenario, dispatches
```

`scenarios/index.ts` is regenerated by `forge generate` from the contents of `scenarios/` (similar to how the mocks barrel file is generated today). User scenario files are *not* touched by the generator after creation; the registry just re-exports them.

### Scenario API

```ts
// src/mocks/scenario-types.ts (generated, do not edit)
import type { Transport } from "@connectrpc/connect";

export interface Scenario {
  /** Unique name; matches the ?scenario= URL param. */
  name: string;
  /** Optional human-readable description (used in dev overlay tooltips). */
  description?: string;
  /**
   * Per-RPC overrides keyed by `${serviceTypeName}/${methodName}`.
   * Anything not listed falls through to the base fixture transport.
   * Throwing a ConnectError surfaces as query.isError in React Query.
   */
  handlers: Record<string, (req: unknown) => unknown | Promise<unknown>>;
  /**
   * Side-effects run once before the first render — for non-RPC state
   * (localStorage flags, URL search params other than `scenario`,
   * sessionStorage). Keep this small. If you reach for `setup()` to fake
   * server data, write a `handler` instead.
   */
  setup?: () => void;
}

export function defineScenario(s: Scenario): Scenario {
  return s;
}
```

### User scenario file

```ts
// src/mocks/scenarios/github-connected.ts (user-authored)
import { create } from "@bufbuild/protobuf";
import { GetGitCredentialResponseSchema } from "@/gen/.../git_credential_pb";
import { defineScenario } from "../scenario-types";

export default defineScenario({
  name: "github-connected",
  description: "User signed in, GitHub credential with repo scope",
  handlers: {
    "controlplane.v1.GitCredentialService/GetGitCredential": (req) =>
      create(GetGitCredentialResponseSchema, {
        provider: req.provider,
        hasToken: true,
        scopes: "user:email repo",
      }),
  },
});
```

### Generated mock transport

```ts
// src/lib/mock-transport.ts (AUTO-GENERATED — DO NOT EDIT)
import type { Transport } from "@connectrpc/connect";
import { ConnectError, Code } from "@connectrpc/connect";
import * as scenarios from "@/mocks/scenarios";

const requested = new URLSearchParams(globalThis.location?.search ?? "").get("scenario");
const active =
  (requested && scenarios.byName(requested)) ?? scenarios.default;

// `setup()` runs once at module init — before any RPC fires.
active.setup?.();

export function createMockTransport(): Transport {
  return {
    async unary(service, method, _signal, _timeoutMs, _header, message) {
      const key = `${service.typeName}/${method.name}`;
      const handler = active.handlers[key];
      if (handler) {
        const result = await handler(message);
        return { message: result, header: new Headers(), trailer: new Headers() };
      }
      // Fall through to the base fixture dispatch (existing behavior).
      return baseFixtureTransport.unary(service, method, _signal, _timeoutMs, _header, message);
    },
    stream() {
      throw new ConnectError("scenarios don't support streaming yet", Code.Unimplemented);
    },
  };
}
```

### CLI command

`forge add scenario <name>` (in `cli/add_scenario.go`) scaffolds:
- `src/mocks/scenarios/<name>.ts` from a template with the right imports and a stub handler
- Regenerates `src/mocks/scenarios/index.ts` so the new scenario is picked up

Flags:
- `--frontend <name>` — disambiguate when the project has multiple frontends
- `--from <existing>` — copy an existing scenario as a starting point

### Dev-mode overlay (optional — separate task)

A `<ScenarioSwitcher>` corner dropdown rendered when `import.meta.env.DEV && import.meta.env.VITE_MOCK_API === "true"`. Lists all scenarios from the registry; selecting one navigates to `?scenario=<name>` and reloads. Not required for agent-driven workflows but useful for human debugging.

## Rules (must be enforced by tests + linter)

- **A scenario file's default export is a `Scenario` (the `defineScenario` return).** Static analysis can verify this; tests will assert on it.
- **`setup()` cannot make network calls.** It runs synchronously before the transport is mounted.
- **Handlers receive proto messages, return proto messages.** No `as any`, no JSON blobs.
- **The default scenario must have empty handlers and no setup.** It's the "nothing overridden" fallback; new scenarios are additive.

## Streaming RPCs

A streaming scenario handler returns an array, iterable, async iterable, or a Promise resolving to any of those. The mock transport adapts whatever you return into the `AsyncIterable<Response>` Connect's `StreamResponse.message` expects.

```ts
// Server-streaming: yield tokens with realistic pacing
"demo.v1.ChatService/StreamCompletion": async function*(req) {
  for (const token of "Hello, world!".split("")) {
    yield create(ChatTokenSchema, { text: token });
    await new Promise(r => setTimeout(r, 30));
  }
},

// Server-streaming: just an array if you don't care about timing
"demo.v1.LogService/Tail": () => [
  create(LogLineSchema, { msg: "first" }),
  create(LogLineSchema, { msg: "second" }),
],

// Client-streaming: the handler input is an array of request messages
"demo.v1.UploadService/UploadChunks": (chunks) => [
  create(UploadResponseSchema, { received: (chunks as Chunk[]).length }),
],
```

If no scenario handler matches a streaming RPC, the transport rejects with `Code.Unimplemented` — base fixtures intentionally don't generate streaming responses because there's no canonical "list of N messages" shape for an arbitrary streaming RPC.

## Stateful scenarios

Scenario files are plain JS modules; module-scope state persists across handler calls until a full page reload. This subsumes sequenced responses, click-driven mutation→list interplay, and polling counters — no framework needed. See `frontend/scenarios` SKILL.md for canonical patterns.

## Limitations and Non-Goals

- **No multi-scenario composition in v1.** A request can match exactly one scenario; later we can support `?scenario=a,b` if a real use case shows up.
- **Auth is still mocked by the stub auth provider.** Scenarios don't touch auth — that's already handled by `VITE_MOCK_API=true`.
- **No automatic state reset on `?scenario=` navigation.** State resets on full page reload only. This is intentional — a scenario file = one stateful session.

## Considered Alternatives

1. **State-seeding (write to Zustand / React Query at boot).** Rejected: skips the parser/serialization code path that's the most common source of UI bugs. Would have hidden both regressions from this session's debug log.
2. **MSW.** Plausible; works at the fetch layer. Rejected because we already have a Connect-aware mock-transport substrate; adding MSW would duplicate the dispatch layer. Scenarios use the substrate that already exists.
3. **Per-test fixtures (no shared scenarios).** Rejected: the same bug is usually hit by Chrome MCP exploration AND by an integration test; sharing the scenario means one fixture maintains both.

## Migration

Pure addition. Existing projects without scenarios continue to work — the mock transport falls through to base fixtures whenever no `?scenario=` is present. The `default` scenario is auto-generated and is empty.
