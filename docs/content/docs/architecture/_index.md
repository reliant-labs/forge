# Forge Architecture

Forge is built on a dual-contract thesis: **proto contracts for external boundaries, Go interface contracts for internal boundaries**. Proto definitions are the source of truth for APIs and configuration. Go interfaces defined in `contract.go` files are the source of truth for internal package boundaries. This separation exists because proto cannot express channels, complex Go types, factories, or variadic arguments -- the things internal packages need.

## Why Two Contract Systems?

Most frameworks treat proto files as the only contract system. Forge recognizes that external and internal boundaries have different needs:

**External boundaries need proto** because they cross process/language/network boundaries. Proto gives you wire format, code generation in multiple languages, and backward compatibility checking. Service RPCs and config schemas belong here.

**Internal boundaries need Go interfaces** because they stay within a single process. Go interfaces can express channels, complex types, closures, and other constructs that proto cannot. Internal packages like email senders, cache layers, and business logic modules use Go interface contracts in `contract.go`.

**Contract enforcement is automatic.** The `forge lint --contract` command scans all service and package types and rejects any exported method that doesn't map to its contract -- whether that contract is a proto service definition or a Go interface.

**LLM-generated code is constrained.** When an AI writes service code, it must implement a contract-defined interface. It can't invent new public methods, change wire formats, or break the contract with other services.

## Single Binary Design

Each project produces a **single Go binary** with a Cobra CLI:

```bash
myproject server                    # Start all services
myproject server api users          # Start only named services
myproject version                   # Print build info
```

There are no per-service binaries, no per-service `cmd/` directories, no per-service Dockerfiles. Services are handler packages that register their Connect RPC handlers onto a shared HTTP mux.

## Explicit Wiring

All construction and dependency injection happens in `pkg/app/wire.go`, which is generated by `forge generate`. There are no `init()` functions, no global registries, and no import side effects.

```go
// pkg/app/wire.go — GENERATED
package app

func BuildDeps(cfg *configv1.Config) (*Deps, error) {
    pool, err := pgxpool.New(ctx, cfg.DatabaseUrl)
    if err != nil {
        return nil, err
    }

    logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

    emailSvc := email.New(email.Deps{Logger: logger})

    return &Deps{
        DB:     pool,
        Logger: logger,
        Email:  emailSvc,
    }, nil
}
```

The config proto is the executable's instantiation contract: it drives what gets constructed, wired, and registered. Mock swapping happens through config -- test configs default to mocks, same wiring code runs.

## The Two Contract Categories

### `proto/services/` — Service RPCs (External)

Service definitions live here:

```
proto/services/
├── api/v1/api.proto
├── users/v1/users.proto
└── orders/v1/orders.proto
```

These protos define the Connect RPC service interfaces. `buf generate` produces Go server and client stubs.

### `internal/<name>/contract.go` — Go Interfaces (Internal)

Internal packages define their contracts as Go interfaces:

```go
// internal/email/contract.go
package email

type Contract interface {
    Send(ctx context.Context, to, subject, body string) error
    SendBatch(ctx context.Context, messages []Message) <-chan Result
}
```

Generated mocks and middleware wrappers are derived from these interfaces via Go AST analysis.

## Generated Code Structure

`forge generate` produces code in the `gen/` directory, which is a separate Go module (using `go.work`). Generated code is never edited by hand but is committed to the repository for reproducible builds.

```
gen/
├── go.mod
├── go.sum
├── services/
│   ├── users/v1/
│   │   ├── users.pb.go           # Proto messages
│   │   └── usersv1connect/
│   │       └── users.connect.go  # Connect RPC server/client stubs
│   └── orders/v1/
│       └── ...
└── forge/options/v1/
    └── ...
```

## Constructor Injection

Services and internal packages receive dependencies through typed `Deps` structs:

```go
type Deps struct {
    DB     *pgxpool.Pool
    Logger *slog.Logger
    Email  email.Contract  // Go interface from internal/email/contract.go
}

func New(deps Deps) *Service {
    return &Service{deps: deps}
}
```

## The Contract Enforcement Linter

The `forge lint --contract` command enforces boundary compliance:

```go
// Passes — GetUser is defined in the proto service
func (s *Service) GetUser(ctx context.Context, req *connect.Request[...]) (*connect.Response[...], error) { ... }

// Fails — not in the proto or contract interface
func (s *Service) GetUserByEmail(email string) (*User, error) { ... }

// Passes — unexported helpers are allowed
func (s *Service) getUserByEmail(email string) (*User, error) { ... }
```

Run the linter:

```bash
forge lint --contract
forge lint --contract ./services/users
```

## Test Harness

Generated test helpers in `pkg/app/testing.go` wire up services with mock dependencies:

```go
func TestUsersService(t *testing.T) {
    svc, mocks := app.NewTestUsersService(t)

    mocks.Email.On("Send", mock.Anything, "user@test.com", mock.Anything, mock.Anything).Return(nil)

    resp, err := svc.CreateUser(ctx, connect.NewRequest(&usersv1.CreateUserRequest{
        Email: "user@test.com",
    }))
    require.NoError(t, err)
}
```

## Design Principles

**Connect RPC everywhere.** Not gRPC. Connect gives HTTP/1.1 + HTTP/2 + gRPC compatibility without gRPC-Web proxies.

**KCL for Kubernetes.** Not Helm, not Kustomize. Type-safe manifest generation with render lambdas.

**No heavy ORM.** Migration-first. SQL migrations are the source of truth. Thin generated CRUD. Complex queries via sqlc.

**Go workspaces.** Generated code in a separate `gen/` module via `go.work`.

**Single Dockerfile.** Multi-stage, BuildKit caches, non-root (`USER 65532:65532`), stripped binaries, `CGO_ENABLED=0`.

**No `init()` registration.** All wiring is explicit, generated in `pkg/app/wire.go`.
