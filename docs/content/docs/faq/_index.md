# Frequently Asked Questions

## General

### What is Forge?

Forge is a proto-first, LLM-optimized application scaffolding framework designed for building microservices with maximum type safety and minimal boilerplate. It generates production-ready Go services and Next.js web applications with CI/CD, Docker, Kubernetes (via KCL), and a proto-enforced architecture using Connect RPC.

### Why proto-first?

Proto-first development ensures:
- Single source of truth for external APIs and config
- Automatic code generation
- Type safety across services via Connect RPC
- Language-agnostic interfaces
- Compile-time validation

### What makes Forge LLM-friendly?

- Clear proto contracts LLMs can understand
- Go interface contracts for internal package boundaries
- Consistent patterns across all services
- Automatic code scaffolding
- Comprehensive documentation

## Development

### How do I create a new service?

```bash
# 1. Define proto under proto/services/<name>/v1/
# 2. Add handler
forge add service <name>
# 3. Generate wiring
forge generate
# 4. Implement service methods in services/<name>/handlers.go
```

See [Creating Services]({{< ref "../guides/creating-services" >}})

### How do I create an internal package?

```bash
# Creates internal/<name>/ with contract.go, service.go, mock, middleware wrapper
forge package new <name>
```

Internal packages use Go interface contracts (`contract.go`), not proto definitions.

### Do I need to write SQL?

For database schema, yes — Forge is migration-first. SQL migrations are the source of truth for schema evolution. The forge-orm library (included in this repo at `pkg/orm/`) introspects the migrated schema and generates thin CRUD code.

### How do I add middleware?

Middleware is implemented as Connect RPC interceptors:

```go
func LoggingInterceptor() connect.UnaryInterceptorFunc {
    return func(next connect.UnaryFunc) connect.UnaryFunc {
        return func(ctx context.Context, req connect.AnyRequest) (connect.AnyResponse, error) {
            log.Printf("calling %s", req.Spec().Procedure)
            return next(ctx, req)
        }
    }
}
```

Interceptors are applied in `cmd/server.go` when building the HTTP handler.

See [Middleware]({{< ref "../architecture/middleware" >}})

### Can I use REST instead of Connect RPC?

Connect RPC supports HTTP/1.1 + HTTP/2 + gRPC wire compatibility on the same endpoint. Clients can use JSON over HTTP/1.1 (REST-like), the Connect protocol, or gRPC wire format — all without proxies.

## Architecture

### How do services communicate?

All services run in a single binary. Within the binary, services communicate via Connect RPC clients injected through `Deps` structs. External communication also uses Connect RPC.

See [Service Communication]({{< ref "../guides/service-communication" >}})

### What is explicit wiring?

`pkg/app/wire.go` is generated by `forge generate`. It constructs all dependencies, builds `Deps` structs, and wires everything together. There are no `init()` functions, no global registries, and no import side effects.

See [Explicit Wiring]({{< ref "../architecture/registry" >}})

### How does middleware work?

Middleware wraps service methods as Connect RPC interceptors to provide cross-cutting concerns like logging, auth, and metrics.

See [Middleware Architecture]({{< ref "../architecture/middleware" >}})

### What are the two contract systems?

- **Proto contracts** for external boundaries: config and API protos define the wire contract via Connect RPC
- **Go interface contracts** for internal boundaries: `contract.go` defines a Go interface that IS the package's contract — supports channels, complex types, factories, and other things proto cannot express

## Database

### Which databases are supported?

Currently PostgreSQL via forge-orm (included in this repo at `pkg/orm/`) and `golang-migrate`. Support for MySQL and SQLite is planned.

### How do I run migrations?

```bash
# Check status
task db:status

# Run migrations
task db:migrate

# Rollback
task db:rollback
```

### Can I use raw SQL?

Yes. SQL migrations are the source of truth for schema evolution. The forge-orm library (`pkg/orm/`) generates thin CRUD code from the migrated schema.

## Testing

### How do I write tests?

Use the generated test harness in `pkg/app/testing.go`:

```go
func TestCreateUser(t *testing.T) {
    svc := app.NewTestUserService(t)
    // svc is wired with mock dependencies from test config
    resp, err := svc.CreateUser(ctx, req)
    // ...
}
```

See [Testing Strategies]({{< ref "../guides/testing-strategies" >}})

### How do I generate mocks?

Mocks are generated automatically:
- Proto API service mocks: `forge generate` creates `services/mocks/`
- Internal package mocks: `forge package new` creates `internal/<name>/mock_gen.go`

### What's the recommended test coverage?

Aim for 80%+ coverage with focus on critical paths.

## Deployment

### How do I deploy to production?

```bash
# Build Docker image (single binary)
docker build -f deploy/Dockerfile -t myapp .

# Deploy to Kubernetes via KCL
kcl run deploy/kcl/envs/prod/main.k | kubectl apply -f -
```

### How do I configure different environments?

Use per-environment KCL files:
- `deploy/kcl/envs/dev/main.k`
- `deploy/kcl/envs/staging/main.k`
- `deploy/kcl/envs/prod/main.k`

The config proto drives runtime configuration via environment variables.

### Does Forge support Docker?

Yes. A single `deploy/Dockerfile` builds the binary using multi-stage builds with BuildKit caches, non-root user (`USER 65532:65532`), and optimized flags.

## Troubleshooting

### Why is my build failing?

Common causes:
1. Missing dependencies: `go mod tidy`
2. Proto syntax errors: `buf lint`
3. Outdated generated code: `forge generate`
4. Contract violations: `forge lint --contract`

See [Troubleshooting]({{< ref "../troubleshooting" >}})

### Why can't I access my service?

Check:
1. Service is running: `<binary> server [services...]`
2. Port is correct (default 8080)
3. Service is included in the server command arguments
4. Wiring is up to date: `forge generate`

### How do I debug issues?

1. Check logs from the running binary
2. Run `forge lint --contract` to verify contracts
3. Ensure `forge generate` has been run after proto changes
