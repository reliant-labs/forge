# Architecture

This guide explains the design decisions behind Forge: the single binary model, how two contract systems enforce boundaries, explicit wiring, and how the generated project structure works.

## Core Thesis

Forge enforces two types of contracts depending on boundary type:

- **Proto contracts for external boundaries**: `proto/config/` and `proto/services/` protos. Config drives instantiation (env vars, flags, bootstrap). API defines external-facing RPC services via Connect RPC.
- **Go interface contracts for internal boundaries**: `internal/<name>/contract.go` defines a Go interface that IS the package's contract. Richer than proto -- supports channels, complex types, factories, variadic args. Generated mocks and middleware wrappers from Go AST.

This constraint exists primarily to control LLM-generated code -- by forcing all public interfaces through contracts, every seam gets automatic logging, tracing, and mocking with no additional effort.

## Single Binary Design

Each Forge project produces a **single Go binary** with a Cobra CLI:

```bash
myproject server                    # Start all registered services
myproject server api users          # Start only named services
myproject version                   # Print build info
```

There are no per-service binaries, no per-service `cmd/` directories, no per-service Dockerfiles. One binary, one Dockerfile, one deployment unit. Services are handler packages that register their Connect RPC handlers onto a shared HTTP mux.

### Why Single Binary?

- **Simpler builds**: One `go build`, one Docker image, one CI pipeline
- **Explicit dependencies**: All wiring visible in one place (`pkg/app/wire.go`)
- **Selective startup**: The `server [services...]` command selects which handlers to register
- **Shared middleware**: One interceptor chain for all services

## Explicit Wiring

All service and package construction happens in `pkg/app/wire.go`, which is generated by `forge generate`. There are no `init()` functions, no global registries, and no import side effects.

```go
// pkg/app/wire.go — GENERATED by forge generate
package app

func BuildDeps(cfg *configv1.Config) (*Deps, error) {
    // Infrastructure from config
    pool, err := pgxpool.New(ctx, cfg.DatabaseUrl)
    if err != nil {
        return nil, err
    }

    logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

    // Internal packages
    emailSvc := email.New(email.Deps{
        Logger: logger,
    })

    // Build the deps object
    return &Deps{
        DB:     pool,
        Logger: logger,
        Email:  emailSvc,
    }, nil
}
```

The config proto is the executable's instantiation contract: it drives what gets constructed, wired, and registered.

## Constructor Injection

Services and internal packages receive dependencies through typed `Deps` structs:

```go
// services/users/service.go
package usersservice

type Deps struct {
    DB     *pgxpool.Pool
    Logger *slog.Logger
    Email  email.Contract  // Go interface from internal/email/contract.go
}

type Service struct {
    deps Deps
}

func New(deps Deps) *Service {
    return &Service{deps: deps}
}

func (s *Service) Name() string {
    return "UsersService"
}

func (s *Service) Register(mux *http.ServeMux, opts ...connect.HandlerOption) {
    path, handler := usersv1connect.NewUsersServiceHandler(s, opts...)
    mux.Handle(path, handler)
}
```

Mock swapping happens through config -- test configs default to mocks, production configs supply real implementations. The same wiring code runs in both cases.

## Two Contract Systems

### Proto Contracts (External Boundaries)

Service definitions live in `proto/services/`:

```
proto/services/
├── api/v1/api.proto
├── users/v1/users.proto
└── orders/v1/orders.proto
```

These protos define Connect RPC service interfaces. `buf generate` produces Go server and client stubs, and `forge generate` creates the service scaffolding.

Config protos in `proto/config/v1/` define the executable's instantiation contract -- what environment variables, flags, and bootstrap parameters the binary accepts.

### Go Interface Contracts (Internal Boundaries)

Internal packages define their contracts as Go interfaces in `contract.go`:

```go
// internal/email/contract.go
package email

type Contract interface {
    Send(ctx context.Context, to, subject, body string) error
    SendTemplate(ctx context.Context, to string, tmpl Template, data any) error
}
```

This is richer than proto -- you can use channels, complex Go types, factories, variadic arguments, and other constructs that proto cannot express. Generated mocks and middleware wrappers are derived from these interfaces via Go AST analysis.

Create internal packages with:

```bash
forge package new email
```

This creates:
- `internal/email/contract.go` -- the Go interface
- `internal/email/service.go` -- implementation scaffold
- `internal/email/mock_gen.go` -- generated mock
- `internal/email/middleware_gen.go` -- generated logging/tracing wrapper

## Contract Enforcement

The `forge lint --contract` command scans all service and package types and enforces boundary rules:

**For proto API services:** All exported methods on the service type must correspond to proto-defined RPCs. You can't accidentally create a public API endpoint that bypasses the contract.

**For internal packages:** All exported methods on the service type must be declared in the `contract.go` interface. Exported types, functions (non-methods), constants, and variables are unrestricted -- only methods on the service type are checked.

```go
// Passes — GetUser is defined in the proto service
func (s *Service) GetUser(ctx context.Context, req *connect.Request[...]) (*connect.Response[...], error) { ... }

// Fails — not in the proto or contract interface
func (s *Service) GetUserByEmail(email string) (*User, error) { ... }

// Passes — unexported helpers are allowed
func (s *Service) getUserByEmail(email string) (*User, error) { ... }

// Passes — exported functions (not methods) are unrestricted
func NewFromConfig(cfg Config) *Service { ... }
```

Run the linter:

```bash
forge lint --contract
forge lint --contract ./services/users  # Specific path
```

## Test Harness

Generated test helpers in `pkg/app/testing.go` provide `NewTestXxx` and `NewTestXxxServer` functions:

```go
func TestUsersService(t *testing.T) {
    // NewTestUsersService wires up the service with mock dependencies
    svc, mocks := app.NewTestUsersService(t)

    // Configure mock behavior
    mocks.Email.On("Send", mock.Anything, "user@test.com", mock.Anything, mock.Anything).Return(nil)

    // Test the service directly
    resp, err := svc.CreateUser(ctx, connect.NewRequest(&usersv1.CreateUserRequest{
        Email: "user@test.com",
    }))
    require.NoError(t, err)
    assert.NotEmpty(t, resp.Msg.Id)
}

func TestUsersServiceE2E(t *testing.T) {
    // NewTestUsersServiceServer starts an HTTP server with the service
    client, cleanup := app.NewTestUsersServiceServer(t)
    defer cleanup()

    // Test via the Connect client
    resp, err := client.CreateUser(ctx, connect.NewRequest(&usersv1.CreateUserRequest{
        Email: "user@test.com",
    }))
    require.NoError(t, err)
}
```

## Directory Layout

A typical project:

```
myproject/
├── cmd/                           # Single binary (Cobra CLI)
│   ├── main.go
│   ├── server.go                  # server [services...] command
│   └── version.go
├── proto/                         # Proto definitions (external contracts)
│   ├── config/v1/                 # Config proto — instantiation contract
│   ├── services/                  # Service RPC definitions
│   │   ├── api/v1/api.proto
│   │   └── users/v1/users.proto
│   └── db/                        # Database entity definitions (deprecated, ORM-owned)
├── gen/                           # Generated code (separate Go module)
├── services/                      # Service handler packages
│   ├── api/
│   │   ├── service.go             # New(deps), Register(mux, opts), Name()
│   │   └── handlers.go            # RPC implementations
│   ├── users/
│   └── mocks/                     # Generated mocks for API services
├── internal/                      # Internal packages (Go interface contracts)
│   └── email/
│       ├── contract.go            # Go interface — THE contract
│       ├── service.go             # Implementation
│       ├── mock_gen.go            # Generated mock
│       └── middleware_gen.go      # Generated wrapper
├── pkg/
│   ├── app/
│   │   ├── wire.go                # GENERATED — constructs all boundaries
│   │   ├── wire_test.go           # GENERATED — test config with mocks
│   │   └── testing.go             # GENERATED — test helpers
│   └── middleware/                # Connect interceptors
├── deploy/
│   ├── Dockerfile                 # Single multi-stage Dockerfile
│   ├── kcl/                       # KCL manifests
│   ├── docker-compose.yml
│   └── k3d.yaml
├── db/migrations/                 # SQL migrations (source of truth)
├── frontends/                     # Next.js frontends (if any)
├── .github/workflows/             # CI/CD
├── forge.yaml
├── buf.yaml / buf.gen.yaml
├── go.work / go.mod
└── Taskfile.yml
```

## Design Principles

**Connect RPC everywhere.** Not gRPC. Connect gives HTTP/1.1 + HTTP/2 + gRPC compatibility without gRPC-Web proxies. Use `connectrpc.com/connect` for Go, `@connectrpc/connect` for TypeScript.

**KCL for Kubernetes.** Not Helm, not Kustomize. KCL provides type-safe manifest generation with a render lambda pattern.

**No heavy ORM.** Migration-first database workflow. SQL migrations are the source of truth. Generated thin CRUD from proto DB entities. Complex queries use sqlc.

**Go workspaces.** Generated code lives in a separate `gen/` module, linked via `go.work`.

**Single Dockerfile.** Multi-stage, BuildKit caches, non-root (`USER 65532:65532`), stripped binaries with `-ldflags="-s -w"`, `CGO_ENABLED=0`.

**No `init()` registration.** All wiring is explicit, generated code in `pkg/app/wire.go`. No import side effects.
