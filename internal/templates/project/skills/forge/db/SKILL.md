# Database Work

All `forge db migrate` subcommands require `--dsn`. Dev DSN:

```
postgres://postgres:postgres@localhost:5432/<project>?sslmode=disable
```

## I need to change the schema

1. Scaffold a migration:
   ```
   forge db migration new <name>
   ```
2. Write both the `.up.sql` and `.down.sql` in the generated pair.
3. Apply it:
   ```
   forge db migrate up --dsn "$DATABASE_URL"
   ```
4. Rebuild generated code:
   ```
   forge generate
   ```

## I need to add or update a query

1. Write or edit SQL in `db/queries/`.
2. Regenerate sqlc types:
   ```
   forge generate
   ```

## I need to check what's in the database

- Inspect the live schema:
  ```
  forge db introspect --dsn "$DATABASE_URL"
  ```
- Or open an interactive shell: `task dev-psql`

## I need to sync proto DB entities

- Generate proto entities from the migrated schema:
  ```
  forge db proto sync-from-db --dsn "$DATABASE_URL"
  ```
- Validate proto entities match the schema:
  ```
  forge db proto check --dsn "$DATABASE_URL"
  ```

## Something went wrong with a migration

1. Check current state:
   ```
   forge db migrate status --dsn "$DATABASE_URL"
   ```
2. If the migration table is stuck (dirty), force to the last good version:
   ```
   forge db migrate force <version> --dsn "$DATABASE_URL"
   ```
3. Fix the SQL and roll forward — don't try to re-run the broken migration.

## Auto-Generated Features

These are handled automatically by `forge generate` based on proto entity conventions:

- **Tenant scoping** — Entities with an `org_id` or `tenant_id` field get automatic `WHERE` clause scoping. See the auth skill for multi-tenant config.
- **Soft delete** — Entities with a `deleted_at` field (or `entity_options.soft_delete: true`) use `SET deleted_at = NOW()` instead of `DELETE`, and List/Get excludes soft-deleted rows.
- **Seed data** — Deterministic SQL seeds in `db/seeds/0002_*.sql` and JSON fixtures in `db/fixtures/` are generated from proto entities. Put custom seed data in `db/seeds/0001_*.sql`.

## Rules

- Migrations are append-only. Never edit a merged migration — write a new one.
- Always write the `.down.sql` (test harness uses it).
- Never `forge db migrate down` against staging/prod — roll forward only.
- Don't hand-edit generated sqlc code.
- Keep seed data out of migrations — use `db/seeds/`.
- Seed files and fixtures are regenerated by `forge generate` — put custom seed data in `db/seeds/0001_*.sql`.
